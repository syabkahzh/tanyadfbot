## 2024-05-14 - SQLite Database Performance Optimization
**Learning:** Adding an index on frequently queried columns in SQLite (like `pending_confirmations(brand)`) can yield significant performance improvements, turning an O(n) scan into an O(1) or O(log n) lookup. However, running any scripts that connect to SQLite may generate binary database files (e.g. `.db-shm`, `.db-wal`) that should NOT be committed to the repository to avoid cluttering version control.
**Action:** Always check `git status` after running local scripts and explicitly unstage/remove any generated SQLite auxiliary binary files before creating a commit.
## 2025-02-12: SQLite Batched Update Performance Fix
* **Problem:** `time_mention_job` exhibited significant performance overhead by executing `UPDATE messages SET time_alerted=1 WHERE id=?` individually inside a loop iterating over potentially thousands of records. This "N+1 query" pattern for updates was scaling linearly with matching messages, taking over ~2 seconds for 500 rows.
* **Solution:** We transitioned from individual row updates to accumulating the affected `id`s into a single `all_ids` list during the loop. We then perform a batched update `UPDATE messages SET time_alerted=1 WHERE id IN (?, ?, ...)` using the accumulated IDs chunked into groups of 900 to ensure we don't hit SQLite's max variable limit (`SQLITE_MAX_VARIABLE_NUMBER`).
* **Performance Impact:** For a test of 500 records, the execution time went from ~2.12 seconds down to ~0.017 seconds, a ~124x speedup. For 5000 records, it completes in around ~2.2 seconds.
